> 将模糊的产品想法，转化为像素级精准的 UI 规格与深度 PRD 文档。

把你的想法直接丢过来——哪怕是零碎的交互描述、粗糙的页面骨架，或者一句「帮我做个后台系统」——`wsc-ux` 会自动判断场景，**零废话直接出活**，或通过高密度提问快速收敛需求，输出专业 PRD 并无缝对接 `wsc-ui-builder` 开发落地。

---

## 三大核心场景

`wsc-ux` 根据你的输入**自动路由**，无需手动切换：

---

### ⚡ 场景 1：碎片梳理 (Fragment Structuring)
**定位：用户的"脑暴收纳盒"。主打零废话、零追问，直接出活。**

你只需丢过来一段零散的描述，它会自动识别类型并输出对应的结构化规格文档，同时自动补全你没想到的边界条件（如防抖、空态、异常处理）。

**三种自动识别类型：**

| 输入类型 | 输出内容 |
| :--- | :--- |
| **交互/状态片段** | 完整状态机表格（Default / Hover / Active / Disabled / Error 的触发条件与视觉表现），自动补全防抖/节流等边界 |
| **页面/布局片段** | 基于栅格的布局层级架构树，从主视觉焦点到辅助信息的完整视线流 |
| **功能想法片段** | 核心 Happy Path 流程步骤 + 空态/断网/权限不足等异常分支 |

**典型触发：**
> "这个按钮点一下转圈，成功变绿弹出提示，失败就抖一下"
> "做个登录页，要有记住密码，忘记密码跳转，还要考虑报错"

---

### 🎨 场景 2：提示词补全 (Prompt Completion)
**定位：Figma (Make/Stitch) 与 wsc-ui-builder 等生成工具的"最强外脑"。**

当你给出粗糙的线框或一句话需求，它会将其扩写为大模型最易理解的高质量 Prompt。

**🔥 核心执行原则：逻辑锁死，视觉留白**

- ✅ **锁死逻辑层**：清晰描述 DOM 父子包裹层级、真实数据占位（如"折线图，横轴月份，纵轴营收"而非模糊的"图表"）、组件状态联动逻辑、异常状态如何占位。
- ❌ **留白视觉层**：严禁写死具体的 Hex 色值、绝对像素 px、具体的贝塞尔曲线（除非用户硬性指定）。把视觉发挥空间留给底层 UI 生成器。
- 💡 **注入顶级审美约束**：自动追加"基于 4px/8px 严格栅格"、"通过字重和灰度拉开信息层级而非滥用亮色"、"内敛克制的微光/阴影动效"等高级设计指令。

**典型触发：**
> "帮我写个 SaaS 工作台的 Make 提示词"
> "补全这个弹窗的 Stitch 提示词"

---

### 🧠 场景 3：资深专家顾问 (Expert Consultant)
**定位：从 0 到 1 帮你把控方向、深挖业务、防错避坑的架构师。**

**核心机制：高密度组合提问，一次性直击要害。** 废除挤牙膏式的单步追问，AI 会根据方向一次性抛出一组结构化的核心问题，迅速收敛需求边界，随后提供方案对比，最终输出极具专业深度的 PRD。

进入场景 3 后，AI 自动判断走哪条轨道：

#### 模式 A：通用 UX 专家（C端 / 网站 / App）
**底层逻辑**：以尼尔森可用性原则和用户体验地图为核心。

一次性高密度提问直击要害：
1. **定位与痛点**：核心受众是谁？目前他们完成这个任务的最大摩擦力是什么？
2. **核心链路**：用户使用的核心 Happy Path 是怎样的？
3. **商业/体验目标**：首要为了提升转化率、留存率，还是降低认知负荷？
4. *[AI 主动预判]*：如果涉及复杂表单或操作，是否需要加入防错（撤销）机制？

收集完信息后，给出 2-3 个方案对比（MVP 验证版 / 标准版 / 完整版），用户选定后输出完整 C 端 PRD。

---

#### 模式 B：B 端产品架构师（后台 / SaaS / ERP / OA / CRM）
**底层逻辑**：基于大量 B 端知识深度蒸馏。B 端核心是**降本、增效、有序**。不创造业务，而是映射线下流程；以角色驱动，极度看重权限与合规。

**B 端内置认知（指导提问方向与专业判断）：**

- **价值框架**：降本（减人/防错）、增效（提速/聚合）、有序（合规/留痕）
- **认知差异**：客户（买单的老板）≠ 用户（实操员工），两者诉求可能冲突
- **三层业务理解**：行业业务流层 → 系统角色权限层 (RBAC) → 页面字段层（必须从上往下理解，不能只看页面）
- **权限设计 (RBAC)**：用户 → 角色 → 权限（数据权限/操作权限/页面权限），数据权限最复杂（销售只看自己的单，主管看全部）
- **审批流六要素**：谁发起、谁审批、按什么规则（如金额>1万走CFO）、审批什么字段、各角色能做什么（通过/驳回/转交）、怎么通知
- **字段设计**：基础属性、状态字段（草稿→审批中→已完成/已驳回）、关联属性、操作型字段，必须考虑必填校验、空态、报错反馈

一次性高密度提问直击 B 端灵魂：
1. **线下业务映射**：真实线下业务是怎么运转的？这个功能主要为了"降本（减人防错）"还是"增效（提速聚合）"？
2. **角色与权限 (RBAC)**：涉及哪些角色操作？数据权限有何隔离要求？
3. **核心业务流与审批**：数据的状态怎么流转？如涉及审批：谁发起、规则是什么、谁能驳回/转交？
4. **核心实体字段**：哪些基础字段、状态字段、关联字段是必须落地的？

收集完信息后，给出 2-3 个架构方案（轻量 MVP 版 / 完整 RBAC 权限+审批流版 / 含数据看板完整版），用户选定后输出极具深度的 B 端 PRD。

---

### 场景 3 收尾：无缝对接开发

PRD 输出完毕后，自动提供后续选项：

> **「需求文档 (PRD) 已完成。接下来您可以：」**
> - **选项 A**：继续深挖某个复杂模块的业务流或交互细节。
> - **选项 B**：直接开始开发。（AI 将自动调用 `wsc-ui-builder` 技能，将 PRD 直接转化为高质量 UI 与前端代码，无需用户手动输入）

## 安装

将 `wsc-ux` 文件夹放入 Claude Code 的技能目录：
```bash
~/.claude/skills/wsc-ux/
```

## 版本记录

### v9.0（当前）
- 场景 3 全面重构：引入双轨专家模式（通用 C 端 UX + 深度 B 端架构师）
- 废除挤牙膏式单步追问，改为高密度组合提问，大幅提升需求收敛效率
- B 端模式深度蒸馏权限设计(RBAC)、审批流六要素、字段规范等核心知识
- 场景 1 和场景 2 明确"零废话、直接出活"的高效工具定位
- PRD 输出后支持无缝自动调用 `wsc-ui-builder` 进入开发环节

### v8.2
- 引入 B 端顾问/导师模式，支持权限/审批流等专业问答
- 对话式脑暴逻辑优化，强制 AskUserQuestion 弹窗

### v6.0
- 引入场景 3「产品顾问」工作流
- 统一弹窗引导策略

### v2.0
